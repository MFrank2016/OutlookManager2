# 管理后台 UI / UX 整体优化设计

## 背景

当前 `OutlookManager2` 的后台已经从旧静态页面逐步迁移到 `Next.js + shadcn/ui + next-themes`，
功能层面基本可用，但界面与交互仍有几个明显问题：

1. 视觉语言还不够统一。
   - 导航、卡片、表格、弹窗、状态提示的层级感不稳定。
   - 管理面板仍使用 emoji 作为 tab 标识，专业感不足。
2. 明暗主题机制偏脆。
   - `ThemeToggle.tsx` 当前通过 `document.documentElement.classList.contains("dark")` 判断主题。
   - `globals.css` 中存在大量 `.dark body.theme-console ...` 的定向覆盖，维护成本高。
3. 核心页面更像“功能可用”，还没有形成成熟后台应有的任务流。
   - 筛选、搜索、批量操作已经具备，但缺少统一组织方式。
   - 页面之间体验不一致，用户需要反复重新学习。
4. 当前新旧风格混用。
   - 一部分页面已经偏现代控制台风格。
   - 另一部分页面仍保留明显的过渡态痕迹，导致整站割裂。

本次优化目标不是“换一层皮”，而是把后台统一升级成一套真正稳定、专业、友好、便捷的管理控制台。

## 目标

本次设计聚焦五件事：

1. 统一后台视觉语言，提升专业感与可信度。
2. 重构明亮 / 暗色主题体系，解决当前切换机制脆弱、样式不一致的问题。
3. 重构后台 App Shell，让导航、页头、内容区、操作区职责清晰。
4. 优化 `账户管理`、`邮件`、`管理面板` 三个核心页面的任务流。
5. 统一列表、筛选、表格、弹窗、空态、加载态、错误态等基础交互模式。

整体风格定为：

- **现代专业型**
- **企业级可信感 + 精致控制台感 + 高频操作效率**

不追求花哨动效，不做娱乐化视觉，不引入与后台场景不符的营销风格。

## 设计原则

### 1. 专业优先

界面要让用户第一眼感受到“这是一个稳定、可信的管理后台”。

具体体现为：

- 不再使用 emoji 作为核心信息架构图标
- 图标统一使用 Lucide 体系
- 颜色、阴影、边框、圆角、间距收敛为统一规则
- 危险操作、系统配置、数据管理等场景体现明确分级

### 2. 任务流优先

不是把控件摆上去就算完成，而是要围绕“用户下一步要做什么”组织页面。

例如：

- 账户页重点支持“搜索 → 筛选 → 选择 → 批量处理”
- 邮件页重点支持“定位邮件 → 查看内容 → 提取关键信息 → 执行动作”
- 管理面板重点支持“进入模块 → 搜索对象 → 修改配置 / 查看数据 / 执行危险动作”

### 3. 一致性优先

用户不应该在每个页面都重新学习：

- 筛选区长什么样
- 表格怎么批量勾选
- 空数据时会出现什么
- 弹窗按钮放哪里
- 成功 / 失败提示如何出现

所有这些模式都应统一。

### 4. 可维护性优先

这次不继续往 `globals.css` 里堆局部特判，而是把主题、表面层级、状态色统一沉淀为可复用 token。

目标是：

- 后续新页面能直接复用
- 改一处主题映射即可全站生效
- 明暗主题不再依赖大量页面级 patch

## 当前问题归纳

结合当前代码与 live 页面，主要问题如下：

### 1. 主题切换“能用但不稳”

live 页面里主题按钮可以切换，但当前实现存在两个结构性风险：

- `ThemeToggle.tsx` 直接读取 DOM class 决定下一主题
- 全局 CSS 中存在大量 `.dark body.theme-console ...` 覆写

这意味着短期可能没问题，但一旦新增页面、局部组件或第三方样式，就容易出现：

- 某些组件浅色正常、深色失真
- 某些局部颜色对比度不够
- 新组件必须继续写额外 dark patch

### 2. 页面骨架还不够系统化

当前后台已有侧边栏与内容区，但缺少统一的“页面上下文层”：

- 页面标题区职责不明确
- 页面主操作与筛选操作没有形成一致层级
- 移动端与桌面端结构相似但没有完全统一的设计约束

### 3. 管理面板识别方式不够专业

`AdminPage` 当前 tab 使用 emoji + 文案，短期可用，但存在几个问题：

- 视觉调性偏轻，不像正式控制台
- 文本长度与图形风格不统一
- 后续增加模块时扩展性差

### 4. 列表与批量操作体验还不够顺手

以账户管理为代表，当前已经具备：

- 搜索
- 标签筛选
- 状态筛选
- 批量刷新 / 批量删除

但筛选区、数据区、批量动作区之间还没有形成更成熟的操作节奏。用户需要自己理解哪些控件属于“过滤”，哪些属于“执行动作”。

## 整体设计方案

## 一、全局设计系统与主题体系

### 1. 设计 token 分层

前端统一收敛为语义化 token，而不是页面直接绑定具体颜色。

建议至少分成以下层：

- 基础色：品牌蓝、强调色、成功、警告、危险、中性灰
- 语义色：`background`、`foreground`、`muted`、`border`、`card`、`panel`、`accent`
- 状态色：`success`、`warning`、`danger`、`info`
- 表面层级：`surface-0`、`surface-1`、`surface-2`、`surface-3`
- 阴影与描边：弱、标准、强调、浮层

组件只依赖语义 token，不再在业务页面里硬写一套明暗颜色。

### 2. 主题映射方式

统一采用两层映射：

- `:root` 负责明亮主题变量
- `.dark` 负责暗色主题变量

原则：

- 组件消费 token
- 页面不直接写 `.dark xxx` 补丁，除非是极少数例外
- 主题切换仅改变 class / token，不改变业务组件结构

### 3. ThemeToggle 重构

`ThemeToggle` 改为基于 `next-themes` 的稳定语义：

- 使用 `theme` / `resolvedTheme` 决定当前状态
- 切换逻辑直接基于 `resolvedTheme === "dark"`
- 避免手动读取 DOM class 作为主判断源

同时补齐：

- 切换按钮的 tooltip / aria 文案
- 按钮在明亮与暗色下的 hover / focus 一致性
- icon 动画在 `prefers-reduced-motion` 下退化

### 4. 视觉气质

整体采用“现代专业型”后台风格：

- 明亮主题：高可读、低噪声、轻玻璃感，但不过度透明
- 暗色主题：深色表面 + 清晰边界 + 控制好的蓝色强调
- 阴影更克制，重点用层次和边框分区，不靠重阴影堆高级感

## 二、App Shell 重构

后台统一采用稳定的 App Shell：

```text
[Sidebar]
  ├─ 品牌区
  ├─ 主导航
  └─ 用户区 / 快捷动作

[Top Header]
  ├─ 页面标题 / 简介
  ├─ 面包屑（可选）
  └─ 页面主操作

[Content]
  ├─ 页面级工具栏
  ├─ 主数据区
  └─ 浮层 / 弹窗 / 抽屉
```

### 1. 左侧导航

左侧导航承担“全局位置感”，不是单纯链接列表。

保留当前桌面可折叠、移动端抽屉的方向，但统一规范：

- 顶部品牌区：品牌名、系统定位、必要时展示环境标识
- 主导航：只保留一级高频入口
- 底部用户区：用户身份、角色、退出、主题切换

当前导航中的主题切换位置基本合理，但需要统一它与收起按钮、用户区之间的主次关系。

### 2. 顶部页头

为所有核心页面统一引入页头结构，包含：

- 页面标题
- 一句话说明当前页面用途
- 主操作按钮
- 必要时显示辅助操作或统计摘要

这样用户进入页面后，不需要先扫描整页才知道“这里是干什么的”。

### 3. 内容区

内容区统一采用：

- 外层控制台背景
- 内层 panel surface
- 标准化内边距与模块间距

避免每页自己决定容器边距、卡片圆角和背景深浅，减少割裂。

## 三、核心页面改造

## 1. 账户管理

账户管理是最高频主工作区，优先按“操作台”思路重构。

### 页面结构

- 页头：标题、说明、`添加账户`、`批量添加`
- 筛选工具栏：搜索、包含标签、排除标签、状态、刷新、重置
- 批量操作条：选中数量、批量刷新、批量删除、清空选择
- 主结果区：表格 / 空态
- 分页区：页码、页大小、跳页

### 优化重点

1. 把“筛选控件”和“执行动作”分区，避免混在一起。
2. 选中账号后，批量操作条要更明显，而不是让用户自己回忆按钮在哪。
3. 表格支持更清晰的 hover / selected / focus 态。
4. 空态不只显示“未找到账户”，而要给出引导动作。
5. 表格在窄屏下采用横向滚动或信息折叠，不允许直接挤坏布局。

## 2. 邮件页面

邮件页面更适合成熟邮箱后台式的“列表 + 详情”结构。

### 页面结构

- 左侧：邮件列表
- 右侧：邮件详情预览
- 顶部：账号选择、关键筛选、搜索
- 详情区：正文、验证码、附件、快捷操作

### 优化重点

1. 减少用户在“列表”和“详情”之间频繁跳页。
2. 提升验证码、关键信息的可见性，适合当前业务场景。
3. 把发送、删除、复制、测试等动作尽量收敛在上下文附近。
4. 加入更成熟的空态 / 无选中态 / 加载态。

## 3. 管理面板

管理面板是这次专业化改造的重点。

### 模块导航

当前 tab 结构保留，但从 emoji tab 升级为统一图标 + 标签模式：

- 数据表管理
- 用户管理
- 系统配置
- 缓存管理
- 验证码规则

每个模块使用 Lucide 图标，不再使用 emoji。

### 数据表管理

当前数据表采用卡片平铺展示，后续保留“快速总览”的优点，但补齐：

- 搜索表名
- 分类 / 风险分级
- 记录数与最近活跃信息的更清晰表达
- 危险表（如系统配置、用户相关）操作提示更强

当进入某张表的详情视图时，使用标准的数据工作区样式，而不是继续散落在不同卡片里。

### 用户管理 / 配置 / 缓存 / 验证码规则

这些模块统一采用：

- 标题说明
- 顶部操作条
- 结果区
- 模态编辑器 / 抽屉编辑器

让整个管理面板内部也像一个小型控制台，而不是多个独立页面拼接。

## 四、组件层统一规范

### 1. 筛选工具栏

所有数据页统一采用同一结构：

- 左侧：搜索 / 主筛选条件
- 中间：次筛选条件
- 右侧：查询、重置、刷新、导出等动作

收益：

- 用户不用每页重学
- 页面之间切换时认知成本更低
- 后续新页面能直接复用同一套 Toolbar 组件

### 2. 表格

统一以下规则：

- 表头固定层级
- 行 hover 高亮
- 选中态明显但不刺眼
- 危险列 / 敏感列有明确样式
- 空态、加载态、错误态统一
- 移动端不允许直接挤爆布局

### 3. 卡片

数据卡片与功能卡片分别规范：

- 数据卡片：偏信息密度、轻交互
- 功能卡片：偏导航或操作入口

避免“所有东西都长成一样的卡片”，导致用户分不清这是数据摘要还是可点击入口。

### 4. 弹窗 / 抽屉 / 表单

统一规范：

- 标题、副标题、关闭按钮位置
- 主按钮、次按钮、危险按钮顺序
- 表单字段间距
- 错误提示就地显示
- 禁用态、加载态、提交态一致

危险操作默认采用更高一级确认，而不是只弹一次轻量 confirm。

### 5. 系统状态体验

统一四类系统态：

1. loading
   - 使用 skeleton 或上下文 loading，而不是大片空白
2. empty
   - 说明为什么为空，以及下一步可以做什么
3. error
   - 给出明确错误信息与恢复动作
4. success
   - 重要动作给到正反馈，但减少打扰式 toast

## 五、交互细节规范

### 1. Hover / Focus / Active

统一所有可点击元素：

- hover 不做大幅位移
- 仅做颜色、边框、阴影微调
- focus ring 明确可见
- active 状态反馈一致

### 2. 无障碍

重点补齐：

- 所有按钮 / 输入框 / 切换器的 aria 文案
- 键盘可达性
- 表单错误提示与 focus 管理
- 明亮 / 暗色对比度满足常规后台可读性要求

### 3. 动效

动效只用于辅助理解：

- 页面进入
- 弹窗打开
- 卡片 hover
- 主题切换细微反馈

不引入夸张位移和无意义炫技。

## 六、实施顺序

建议按三阶段推进。

### 第一阶段：全局地基

- 主题 token 重构
- ThemeToggle 重构
- App Shell 统一
- 按钮、卡片、基础表格样式统一

### 第二阶段：核心页面

- 账户管理
- 邮件页面
- 管理面板

顺序按使用频率和复杂度推进，先把最高频路径稳定下来。

### 第三阶段：收口与细节

- 弹窗 / 抽屉 / 表单统一
- 空态 / loading / error 统一
- 移动端与窄屏溢出修复
- 全站视觉细节 polish

## 七、风险点

本次改造的主要风险有三个：

1. 旧样式补丁较多，主题收敛后可能暴露更多历史问题。
2. 管理面板数据密度较高，必须防止“更好看但更难用”。
3. 组件复用面广，表格 / 筛选 / 弹窗变更容易引发跨页面回归。

因此实施时需要保证：

- 每改一层就做一次可视验收
- 主题层、布局层、页面层分开推进
- 先建立统一组件，再批量替换业务页面

## 八、验证方案

### 代码层验证

至少覆盖：

- `frontend` lint
- TypeScript 构建 / 类型检查（如仓库已有对应入口）
- 关键组件与关键页面的回归检查

### live 验证

必须做真实页面验收，而不是只看源码：

1. 登录页明亮 / 暗色切换
2. Dashboard 壳层在明暗主题下的一致性
3. 账户页搜索、筛选、批量操作可用性
4. 邮件页列表 / 详情的连续操作体验
5. 管理面板 tab、卡片、表格、弹窗在明暗主题下的表现
6. 桌面与移动窄屏下无明显布局破坏

## 九、验收标准

本次优化完成后，应满足以下结果：

1. 整站视觉语言统一，不再有明显新旧风格割裂。
2. 明亮 / 暗色主题切换稳定，全站主要页面无明显样式漏网。
3. `账户管理`、`邮件`、`管理面板` 的高频操作路径明显更顺手。
4. 表格、筛选、弹窗、空态、提示反馈具备一致交互模式。
5. 后续新增后台页面时，可以复用统一的 shell、toolbar、table、dialog 体系。

## 非目标

本次不做以下事情：

1. 不改后端业务语义和 API 协议，除非为 UI 体验所必需。
2. 不重做整套权限模型。
3. 不引入新的重型 design system 依赖。
4. 不把后台改成营销官网式视觉风格。

## 下一步

设计确认后，进入 implementation 前建议先做两件事：

1. 使用独立 worktree 隔离本轮 UI / UX 改造。
2. 基于本设计文档拆出详细 implementation plan，按“主题 / 壳层 / 页面 / 收口”四个维度执行。
